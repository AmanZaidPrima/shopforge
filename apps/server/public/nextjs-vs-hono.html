The core issue with Next.js here
Even with no React on the client, Next.js still ships its runtime JS (~100kb) to every browser, runs a full React SSR pass on the server for each request, and is locked to Node.js. These are features for React apps — unused weight for a template-rendering server.

Why Hono for this specific stack
This stack: Eta templates from MySQL · Alpine.js · No React anywhere
Hono is a plain HTTP server. It receives a request, your code fetches the template, renders it with Eta, streams the HTML out. There is no rendering abstraction between your code and the response. That is exactly what this architecture needs — and exactly what Next.js adds overhead around.

Bottom line

Both approaches produce the same output — server-rendered HTML with Alpine.js on the client. The difference is what runs in between. Next.js executes a React SSR pipeline and ships its runtime to the browser for a page that has no React components. Hono runs eta.renderString(template, data) and sends the result directly — no unused steps, no idle runtime, no deployment constraints. For this specific stack — Eta from MySQL, Alpine.js, no React — Next.js has no advantage. It is a React framework being used without React.
